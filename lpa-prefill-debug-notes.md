## Key conclusion so far
Several address blocks in the existing filler are mapped to the wrong suffix family. The most clearly confirmed errors are:

| Data | Current wrong mapping | Confirmed correct mapping |
|---|---|---|
| Attorney 1 address | `Address 1_3*` | `Address 1_2*` |
| Certificate provider address | `Address 1_9*` | `Address 1_13*` |

These wrong field names would directly cause the pre-fill failures the user is reporting.

## Visual verification from generated `/api/lpa/1/pdf` output

I opened the actual generated PDF returned by the live endpoint after the latest rewrite.

### Observed results

| Page | Observation |
|---|---|
| Cover / intro pages | PDF downloads successfully and remains viewable |
| Section 1 donor page | Donor title, first name, last name, and email are populated; donor date of birth and donor address are still blank |
| Section 2 attorneys page | Attorney 1 first-names field is populated with the business name and trust-corporation checkbox is ticked, but attorney address lines are still blank |
| Section 2 attorney 2 block | Remains blank in this test output |

### What this proves

The endpoint now generates a valid PDF, but some missing pre-fill is not only a PDF field-name problem. At least part of the issue is now likely due to the shape of the stored LPA data not matching the property names used in `lpaFillPdf.ts`.

Examples:
- Donor address may not be stored as a single `donorAddress` string.
- Attorney addresses may not be stored as `attorneys[i].address`.
- DOB may be stored in a different field or format than `DD/MM/YYYY`.

### Next debugging step

Inspect the actual `lpaRecords` row shape and the JSON structure of `attorneys`, `replacementAttorneys`, and donor/certificate-provider fields returned by the database, then adapt the filler to those real property names.
